## Purpose

A local-first personal dashboard that turns Claude Code session logs into a daily
token burn record ‚Äî a feedback loop for delegated intelligence. The goal is not to
maximize or minimize usage. The goal is to see *what work moved*, whether it was
useful, and what to delegate next.

This is Brett Coryell's personal tool. It will not be shared publicly without scrubbing.

## Data Schema

### Primary data file: `public/data/daily-burn.json`

One JSON array. One object per calendar day (US/Pacific timezone). Generated by the
collector script. This file is committed to the repo (normalized token totals only ‚Äî
no session content, no project names, no personally identifying text).

```typescript
// TypeScript type ‚Äî exact schema, no extra fields permitted
type DayRecord = {
  date: string;                    // "YYYY-MM-DD" in US/Pacific time
  claude_code_input: number;       // integer, non-negative
  claude_code_output: number;      // integer, non-negative
  claude_code_cache_read: number;  // integer, non-negative
  claude_code_cache_create: number;// integer, non-negative
  claude_code_api_requests: number;// integer, non-negative
  claude_code_sessions: number;    // integer, non-negative
  claude_chat_sessions: number;    // integer, non-negative
  claude_chat_est: number;         // integer, non-negative
  total_exact: number;             // integer, non-negative
  total_est: number;               // integer, non-negative
  sources: string[];               // e.g. ["cadence"], ["coda"], ["cadence","coda"]
  driver: string;                  // "" if unannotated
  evidence: string;                // "" if unannotated, max 200 chars
};
```

**Schema is closed.** No field may be added without a spec revision. The collector test
suite asserts that every output row has exactly these 14 fields and no others.

### Field definitions

| Field | Type | Source | Notes |
|---|---|---|---|
| `date` | string | First `record.timestamp` in session file, converted to US/Pacific | "YYYY-MM-DD" |
| `claude_code_input` | int | Sum of `message.usage.input_tokens` across all assistant messages | Non-cached only |
| `claude_code_output` | int | Sum of `message.usage.output_tokens` across all assistant messages | |
| `claude_code_cache_read` | int | Sum of `message.usage.cache_read_input_tokens` | |
| `claude_code_cache_create` | int | Sum of `message.usage.cache_creation_input_tokens` | |
| `claude_code_api_requests` | int | Count of records where `type == "assistant"` AND `message.usage` exists | Not conversational turns |
| `claude_code_sessions` | int | Count of JSONL files whose first timestamp falls in this date | |
| `claude_chat_sessions` | int | From `annotations.json` `claude_chat_sessions` field | Manual entry |
| `claude_chat_est` | int | `claude_chat_sessions √ó CHAT_TOKENS_PER_SESSION` (default: 75000) | Labeled ESTIMATED everywhere |
| `total_exact` | int | `claude_code_input + claude_code_output + claude_code_cache_read + claude_code_cache_create` | Never mixed with est |
| `total_est` | int | `claude_chat_est` | Shown separately, labeled |
| `sources` | string[] | Machine name(s) contributing to this date | e.g. `["cadence"]` |
| `driver` | string | From `annotations.json` `driver` field | Empty string if unannotated |
| `evidence` | string | From `annotations.json` `evidence` field | Max 200 chars; empty string if unannotated |

### Token counting formula (verified empirically 2026-06-09)

```
total_exact = input_tokens + output_tokens + cache_read_input_tokens + cache_creation_input_tokens
```

**Empirical verification:** In Claude Code JSONL, `message.usage.input_tokens` reports
only the non-cached portion of input (e.g., `3` when `cache_read_input_tokens = 14415`).
Cache tokens are in separate fields. There is NO double-counting. This matches
Anthropic's billing model: each field maps to a distinct billing bucket.

Fixture showing verified values:
```json
{
  "input_tokens": 3,
  "cache_read_input_tokens": 14415,
  "cache_creation_input_tokens": 12323,
  "output_tokens": 166
}
// total_exact = 3 + 14415 + 12323 + 166 = 26,907
```

### Timestamp bucketing rule

- **Source field:** Top-level `record.timestamp` on each JSONL record (ISO 8601 UTC).
- **Bucketing unit:** Per session file (not per message).
- **Date assignment:** Use the `record.timestamp` of the **first record in the file** to
  determine the session's date bucket. Convert UTC ‚Üí US/Pacific before date extraction.
- **Midnight-spanning sessions:** If a session starts at 23:55 PT and has messages past
  midnight, all tokens from that session go into the date of the first record. Messages
  are not split across dates.

**Rationale:** "What day did I do this work?" is better answered by when the session
started than by when the last message fired.

### Multi-source aggregation rule

When running the collector against multiple machines' session files:

```bash
make collect          # Cadence's sessions (default)
make collect-iMac     # Adds iMac sessions to the same daily-burn.json
```

The collector uses `(date, source)` as a unique key for ingest. When merging:
- Each machine's tokens are summed into the date row **additively**
- The `sources` field is updated to include the machine name
- Running `collect.py` twice with the same `--machine` flag on unchanged files is idempotent
- Running with a second `--machine` adds that machine's tokens to the date row

If a machine's sessions are re-scanned after new sessions are added, only the
incremental tokens are added (identified by session file hash or session ID).

### Annotations sidecar: `data/annotations.json`

Contains manually-curated fields that the collector merges into `daily-burn.json`.
Committed to the private repo. Strip before any public share.

```json
{
  "2026-06-09": {
    "driver": "infrastructure",
    "evidence": "Mac Mini bootstrap, portability work, MSM v3 closeout",
    "claude_chat_sessions": 2
  }
}
```

**Merge behavior:** Collector always overwrites `driver`, `evidence`, and
`claude_chat_sessions` from `annotations.json` into the corresponding `daily-burn.json`
row. This is the canonical mechanism for updating these fields. Direct edits to
`daily-burn.json` are overwritten on next collect.

## Five Views

### Global controls (affect all views)

**Time range selector:** `30d | 90d | 1y | all` ‚Äî consistent across all five views.
Default: `90d`. Changing this selector updates all five views simultaneously.

**Header KPI cards:**
- `EXACT TOTAL` ‚Äî sum of `total_exact` for range. Label: "measured"
- `SESSIONS` ‚Äî sum of `claude_code_sessions` for range. One JSONL file = one session.
- `EST. CHAT` ‚Äî sum of `claude_chat_est` for range. Label: "floor estimate"

The KPI for total may NOT show a combined exact+estimated total. The two figures must
remain visually separate.

---

### View 1 ‚Äî Daily Burn Heatmap

**Layout:** GitHub contribution-graph style. Weeks as columns, days as rows.
Week starts on **Sunday** (consistent with GitHub and View 2).

**Color logic:**
- Cells colored by `total_exact` using a **logarithmic** color scale.
- Color bins: 0 (empty/dim), 1‚Äì1K, 1K‚Äì10K, 10K‚Äì100K, 100K‚Äì1M, 1M+ (5 non-zero bins).
- A day with 1M tokens and a day with 100K tokens must be visibly distinct colors.
- A day with `total_exact = 0` and `total_est > 0` (pure Ariel day) shows a distinct
  pattern: dim background cell WITH a small dot or accent indicator (`data-estimated="true"`
  on the cell element). This is different from a zero-activity day which is plain dim.
- A zero-activity day (no exact, no estimated) is a plain dim cell.

**Tooltip on hover:** Date, `total_exact` (labeled "measured"), `total_est` (labeled
"estimated", only shown if nonzero), `driver` label (if annotated).

**Responds to:** Global time range selector.

---

### View 2 ‚Äî Weekly Trend Line

**What:** 7-day rolling total. One data point per week. **Week boundary: Sunday.**
(Same Sunday-start as the heatmap ‚Äî the two views must agree on week boundaries.)

**Y-axis:** Logarithmic scale. Peak week labeled directly on the chart with value and date.

**Lines:**
- Exact lane: solid line, `total_exact` aggregated per week.
- Estimated lane: dashed line, `total_est` aggregated per week. Only rendered if
  non-zero data exists in the range.

**Responds to:** Global time range selector. If fewer than 2 weeks in the range,
renders a single point (not an error).

---

### View 3 ‚Äî Burn Drivers

**What:** Horizontal bar chart. One bar per driver. Sorted descending by token share.

**Threshold:** If fewer than **7 days within the currently selected time range** have a
non-empty `driver` field, show placeholder: "Annotate sessions to see drivers."
(Scoped to current range, not global dataset.)

**Each bar:** Driver name + percentage of `total_exact` for the range.

**Hover/click:** Shows `evidence` strings for the top contributing days to that driver.
Truncate evidence at 200 chars (already capped in schema).

**Responds to:** Global time range selector.

---

### View 4 ‚Äî Scale Equivalents

**Source for all calculations:** `total_exact` only. Estimated tokens are NOT included
in scale equivalent calculations. A footnote clarifies: "Exact Claude Code tokens only.
Estimated chat activity not included."

**Cards (show the math, not just the number):**

| Card | Formula |
|---|---|
| Query-equivalents | `total_exact / 1000` |
| Electricity (kWh) | `(total_exact / 1000) √ó 0.00034` |
| Netflix movies (2h) | `electricity_kwh / 0.45` |
| Code volume (LOC) | `total_exact / 15` |
| Engineer-years | `(total_exact / 15) / 10000` |

**Disclaimer (always visible):** "These are scale translations, not measured utility,
billing, or environmental accounting."

**Responds to:** Global time range selector.

---

### View 5 ‚Äî Daily Detail Table

*(Previously titled "Moving Average Table" ‚Äî renamed to avoid confusion. No rolling
average column is planned for v1.)*

**What:** Per-day breakdown table. Most recent date first.

**Columns:** Date | Total Exact | Claude Code | Claude Chat Est | Sessions | API Requests | Driver

**Trust indicators:**
- Claude Chat Est column: every cell with a non-zero value shows `EST` badge.
- Total Exact column: shows `MEASURED` badge in the column header.

**Responds to:** Global time range selector.

## Collector Script (`scripts/collect.py`)

**Input:** `~/.claude/projects/*/` JSONL session files (default; overridable)  
**Output:** `public/data/daily-burn.json` (upsert by date)  
**Merges:** `data/annotations.json` on every run (driver, evidence, claude_chat_sessions)

**Idempotency:** Running twice on unchanged files produces identical output. Idempotency
is guaranteed by the session file hash check ‚Äî already-processed sessions are skipped.

**Timezone:** All date bucketing in US/Pacific (`America/Los_Angeles`).

**Error handling:** Malformed JSONL lines are skipped with a warning to stderr.
`json.JSONDecodeError` is caught per-line. The collector never raises an unhandled
exception from bad input data. Missing or empty JSONL files are skipped silently.

**CLI flags:**

| Flag | Default | Description |
|---|---|---|
| `--sessions-root PATH` | `~/.claude/projects/` | Directory to scan for JSONL files |
| `--machine NAME` | `cadence` | Machine identifier written to `sources[]` |
| `--chat-tokens-per-session N` | `75000` | Override estimate constant for chat sessions |
| `--dry-run` | off | Print what would be written, don't write |
| `--verbose` | off | Print per-session token counts in tab-separated format: `session_id\tdate\tinput\toutput\tcache_read\tcache_create\tapi_requests` |

**Reconciliation output (always printed):**
```
[collect] Scanned 3 session files ‚Üí 1 date (2026-06-09)
[collect] 2026-06-09: exact=5,831,412 est=0 sessions=3 api_requests=12 sources=["cadence"]
[collect] Output written to public/data/daily-burn.json (1 row)
```

## Deployment Model

**Local MVP:**
```
make collect    # Collect Cadence sessions ‚Üí updates daily-burn.json
make dev        # npm run dev ‚Üí localhost:5173
make test       # pytest + playwright (combined)
make build      # npm run build ‚Üí dist/
```

**Crontab (daily auto-collect at 11 PM PT):**
```
0 23 * * * cd /Users/brettcoryell/Code/AI/token-burn && make collect 2>&1 >> ~/.local/share/token-burn/collect.log
```

**Vercel deploy (optional):**
- Push to GitHub (private repo) ‚Üí Vercel auto-deploys from `dist/`
- `daily-burn.json` is committed, so Vercel has the last-committed snapshot
- **Expected staleness:** up to 24 hours (cron commits at 11 PM, Vercel redeploys on push)
- This is acceptable. The dashboard is a daily review tool, not a live monitor.
- Data freshness note shown in footer: "Updated [date of last data row]."

**Cross-machine import (future, Tailscale required):**
```
make collect-iMac
# = rsync iMac sessions ‚Üí /tmp/iMac-sessions/ && python3 scripts/collect.py --sessions-root /tmp/iMac-sessions/ --machine coda
```
